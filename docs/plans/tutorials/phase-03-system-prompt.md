# Phase 3: System Prompt

> Series: Mini-Claude-Code Learning Plan — see [`../phase-breakdown.md`](../phase-breakdown.md) for full context.
> **Prerequisites:** [`phase-01-agent-loop.md`](phase-01-agent-loop.md) and [`phase-02-tool-system.md`](phase-02-tool-system.md). This phase builds directly on the `toolRegistry` Phase 2 left behind — read it first if you haven't. This is the last phase in this initial three-phase run.

## Goal

Phase 1 left `runAgentLoop()` accepting a `systemPrompt?: string` option and doing nothing more with it than forwarding it to `client.messages.create()`'s `system` parameter. Every phase since has left that seam alone — `index.ts` has just been passing a single hardcoded string, `"You are a terse coding assistant."`

By the end of this phase you'll have a `src/prompt.ts` module that *produces* that string instead of hand-typing it: a composed prompt built from separately-maintainable pieces (identity, behavioral rules, a risk framework, tone, output style), a tools section generated by iterating Phase 2's `toolRegistry` instead of a second, hand-written tool list that could quietly drift out of sync with it, an environment section that injects runtime facts (cwd, date, platform, git status) at call time rather than authoring time, and an `@include`-style mechanism for pulling in project-specific instructions from a `CLAUDE.md` file and a `.claude/rules/` directory. `runAgentLoop` itself does not change in this phase — not its signature, not its body, not one line. That's not an incidental detail; it's the entire point (Concept 1).

## Why this is interview material

Ask most engineers "how do you prompt an LLM well" and you'll get generic advice — be specific, give examples, use delimiters. That advice is real but it isn't what this phase is about. Prompting *an agent* is a narrower, harder problem: you're not just steering the shape of one response, you're steering a loop that runs many turns, decides for itself when to call which tool, and has to be trusted with write access to a filesystem. The system prompt is where you encode "what kind of agent is this" — and being able to explain, concretely, how that encoding is engineered (not just written) is what separates "I've used ChatGPT a lot" from "I understand agent architecture."

Three things in this phase are specifically worth being able to say out loud in an interview:

1. **The system prompt is a real separation of concerns, not a formality.** `agent.ts`'s loop treats `systemPrompt` as an opaque string it never inspects. That means you can radically change agent *behavior* — tone, caution level, which tools it prefers, how verbose it is — by editing `prompt.ts` alone, with zero changes to the control-flow code that runs the loop. This phase's Verify section proves this with an actual before/after.
2. **A hand-maintained tool description is a bug waiting to happen — the same lesson Phase 2 taught about schemas, now applied to prose.** Phase 2, Concept 1 warned that a schema and its `execute` function are two independent sources of truth that can silently drift apart. The exact same failure mode exists if you hand-type a paragraph describing your tools into a prompt string: add a tool to the registry, forget to update the prose, and the model now has stale information about what it can do. The fix is identical in spirit: generate the description from the registry, so there's exactly one source of truth.
3. **Structure and ordering are load-bearing, not stylistic.** Real Claude Code's system prompt is not one giant paragraph — it's built from named sections in a specific, deliberate order, and that order encodes a real claim about how LLMs weight earlier context differently from later context. This phase's structure mirrors that reasoning, not just its aesthetics.

## Concept 1: The system prompt is a parameter, not a personality

Here's the framing that resolves a common confusion: it's tempting to think of "giving the agent a system prompt" as something conceptually different from "calling a function with an argument." It isn't. Look at what Phase 1 already built and never needed to touch again:

```typescript
// unchanged since Phase 1 — src/agent.ts
export interface RunAgentLoopOptions {
  client: Anthropic;
  model: string;
  systemPrompt?: string;
  tools?: Anthropic.Tool[];
  maxTokens?: number;
}
```

and inside the loop:

```typescript
// unchanged since Phase 1 — src/agent.ts
const response = await client.messages.create({
  model,
  max_tokens: maxTokens,
  system: systemPrompt,
  tools,
  messages,
});
```

`systemPrompt` is a plain string, passed straight through. `agent.ts` never parses it, never branches on its contents, never even checks whether it's defined (`system: undefined` is a perfectly valid request — the API just omits the parameter). This phase's entire job is to write the *function that produces the string* — `buildSystemPrompt()` in a new `prompt.ts` — and change exactly one call site in `index.ts` to call it instead of hardcoding a literal. Nothing in `agent.ts` changes. If you find yourself wanting to add an `if (systemPrompt.includes(...))` anywhere in the loop, that's a sign the abstraction is leaking — the loop's job is to run turns, not to interpret prompt text.

This is also, not coincidentally, exactly why `systemPrompt` was already typed as a plain `string` (not some richer `SystemPromptConfig` object) all the way back in Phase 1: the loop was deliberately kept ignorant of *how* the string gets built, so that "how it gets built" could become an entirely separate, swappable subsystem later — which is precisely what's happening now.

## Concept 2: Composing from parts — sections, not one giant string literal

The naive version of a system prompt is a single template literal with everything in it — identity, rules, tone, environment info, all run together. That's fine for a demo and actively bad for a maintained project, for the same reason a 2,000-line function is bad: you can't change one concern (say, tone) without re-reading and risking everything else in the same blob.

Real Claude Code's system prompt is explicitly built from **seven named static sections**, assembled in a fixed order (read directly from `claude-code/src/constants/prompts.ts`, confirmed by function names at these exact line numbers):

```
1. getSimpleIntroSection()          → Identity: who am I, what are my hard boundaries
2. getSimpleSystemSection()         → System: operating facts (what's visible to the user, etc.)
3. getSimpleDoingTasksSection()     → Doing Tasks: coding behavior, anti-patterns
4. getActionsSection()              → Actions: risk framework for what needs confirmation
5. getUsingYourToolsSection(...)    → Using Your Tools: tool preferences
6. getSimpleToneAndStyleSection()   → Tone and Style: output formatting
7. getOutputEfficiencySection()     → Output Efficiency: verbosity control
```
— `claude-code/src/constants/prompts.ts`, lines 175–430 (function definitions), assembly order confirmed in `how-claude-code-works/docs/14-system-prompt-design.md`, §13.6 ("组装顺序" / assembly order diagram, lines 3943–3966).

The ordering is not arbitrary. The companion Chinese-language deep-dive states the reasoning directly: *"先建立身份和约束框架，再填充具体行为指导"* — "establish identity and the constraint framework first, then fill in specific behavioral guidance" (`claude-code-from-scratch/docs/03-system-prompt.md`, line 29) — **the concepts a model forms early become the lens it uses to interpret everything that comes after.** Telling a model "you are an interactive coding agent" *before* telling it "don't add unnecessary error handling" gives the second instruction a frame to land in; reversed, the same instruction is just a floating rule with no anchor.

The reverse principle governs where *dynamic*, per-conversation content goes: **at the end**, not the start. `claude-code-from-scratch/docs/03-system-prompt.md` puts memory, skills, and agent descriptions last in its template and explains why in one line: *"{{memory}}、{{skills}}、{{agents}} 放在末尾——近因效应，这些动态内容的权重更大"* — "memory/skills/agents go at the end — recency effect, this dynamic content gets weighted more heavily" (line 149). This phase's environment section (cwd, date, git status — Concept 5) and the `CLAUDE.md`/rules content (Concept 6) both go at the *end* of our composed prompt for the identical reason: they're the parts that change every run, and recency gives them the most weight right when the model is about to start acting.

Our composition mechanism in this phase is the simplest version of "build from parts" that still earns the name: a set of named string constants and small builder functions, joined together by one top-level `buildSystemPrompt()`. This mirrors `claude-code-from-scratch/src/prompt.ts`'s actual structure — a `SYSTEM_PROMPT_TEMPLATE` string plus a handful of functions (`getGitContext()`, `loadClaudeMd()`) that compute the pieces the template's placeholders get filled with (`claude-code-from-scratch/src/prompt.ts`, lines 101–230, read directly). We adapt this slightly: instead of one giant template literal with `{{placeholder}}` tokens for *every* section (including the fully-static ones), we keep genuinely static sections as separate exported constants and only use runtime template-variable substitution (Concept 5) for content that's actually different every run — cwd, date, git status. The static sections don't need placeholder syntax because nothing about them varies; giving them one anyway would just be indirection with no payoff.

## Concept 3: Anti-pattern inoculation and the blast-radius framework

Two content-design techniques are worth understanding by name, because they generalize far past this one prompt.

**Anti-pattern inoculation.** Telling a model "be concise" is a positive instruction, and positive instructions leave room for self-serving interpretation — the model can convince itself that adding a docstring to every function *is* what concise, high-quality code looks like. The fix real Claude Code uses is to state the *negative* explicitly: *"Don't add docstrings, comments, or type annotations to code you didn't change"* (`claude-code/src/constants/prompts.ts`, `getSimpleDoingTasksSection()`, line 201, quoted directly). The deep-dive doc names this pattern explicitly: *"明确告诉模型'不要做什么'，比只描述'要做什么'有效得多"* — "explicitly telling the model what NOT to do is far more effective than only describing what TO do" (`claude-code-from-scratch/docs/03-system-prompt.md`, line 43) — because a negative instruction removes the self-justifying interpretation gap a positive one leaves open. This phase's `DOING_TASKS_SECTION` (Implement 1) keeps this pattern intact: "don't add error handling for scenarios that can't happen," not "write clean code."

**The blast-radius framework.** The naive way to keep an agent from doing something dangerous is to enumerate a list of forbidden actions. That list is always incomplete — the very first novel situation not on the list leaves the model with no guidance at all. Real Claude Code instead teaches a two-dimensional *reasoning framework* the model can apply to situations no one enumerated in advance: *"Carefully consider the reversibility and blast radius of actions"* (`claude-code/src/constants/prompts.ts`, `getActionsSection()`, line 258, quoted directly) — **reversibility × scope of impact.** High risk is the intersection of "hard to undo" and "affects something beyond your own local sandbox" (force-pushing, dropping a database table); low risk is anything both reversible and local (editing a file the user is actively looking at). A model that's internalized the *framework* can reason about a brand-new tool call it's never seen a rule for; a model that only has an enumerated blacklist cannot. `claude-code-from-scratch` explicitly kept this section "完整保留" (kept whole, unsimplified) even in its otherwise-trimmed-down teaching implementation, with the stated reason: *"安全性不能简化"* — "safety is not something you get to simplify" (`claude-code-from-scratch/docs/03-system-prompt.md`, line 299, the "简化取舍" / simplification-tradeoffs table). One more rule rides along with this framework and is easy to miss: *authorization doesn't generalize* — a user approving one push doesn't mean every future push is pre-approved. That line earns its place in the prompt precisely because it's the kind of nuance a blacklist can't express at all.

## Concept 4: The tools section — generated from the registry, not hand-typed

This is the concept this phase adds that isn't just "port the reference prompt" — it's the direct, load-bearing consequence of Phase 2's registry design, and it's worth being precise about what production Claude Code does here versus what this phase deliberately does differently, and why.

**What the API already knows without any help from the prompt text.** Recall Phase 1, Concept 1: every tool's `name`, `description`, and `input_schema` already go over the wire in the `tools` parameter of the Messages API request, completely independent of the `system` parameter. The model does not need a paragraph in the system prompt telling it a tool called `read_file` exists — the `tools` array *is* that information, structured and authoritative. This is why real Claude Code's own `getUsingYourToolsSection()` is short: it isn't re-describing what each tool does (the `tools` array already did that); it's expressing *preferences between tools that overlap in capability* — specifically, steering the model away from reaching for its general-purpose `Bash` tool when a narrower, dedicated tool exists for the same job. Quoting the real function directly (abridged real-source excerpt, `claude-code/src/constants/prompts.ts`, lines 269–314):

```typescript
// abridged real-source excerpt — claude-code/src/constants/prompts.ts
function getUsingYourToolsSection(enabledTools: Set<string>): string {
  const providedToolSubitems = [
    `To read files use ${FILE_READ_TOOL_NAME} instead of cat, head, tail, or sed`,
    `To edit files use ${FILE_EDIT_TOOL_NAME} instead of sed or awk`,
    // ...
  ];
  const items = [
    `Do NOT use the ${BASH_TOOL_NAME} to run commands when a relevant dedicated tool is provided...`,
    providedToolSubitems,
    // ...
  ].filter(item => item !== null);
  return [`# Using your tools`, ...prependBullets(items)].join(`\n`);
}
```

Notice this function takes `enabledTools: Set<string>` as a parameter and references tool names through constants (`FILE_READ_TOOL_NAME`, not the string literal `"Read"`) — it is *itself* generated from what's actually available, not an independently-hand-typed paragraph. That's the same principle this phase applies, just taken one step further.

**Why this phase's version goes further than that.** Two things make our situation different from production's, and both point toward generating a fuller section from `toolRegistry` directly rather than writing short preference bullets:

- **We don't have a `Bash` tool in this registry at all.** Phase 2 built exactly three tools — `read_file`, `edit_file`, `list_files` — with no general-purpose shell-execution tool for the model to be steered away from. Writing "use `read_file` instead of `cat`" the way production does would be prompting the model away from a fallback that doesn't exist in our tool set — actively confusing, not merely redundant. So the "steer away from the generic tool" framing doesn't transfer; what *does* transfer and matters here is simpler: describe what's actually in the registry, generated from the registry, so the description can never go stale.
- **`readOnly` is metadata worth surfacing to the model, and `toolRegistry` is the only place it lives.** Phase 2, Concept 6 set a `readOnly: boolean` on every `ToolDefinition` specifically so later phases could consume it — Phase 5 for concurrency, Phase 6 for permission gating. Nothing in this series has surfaced that flag to the *model itself* yet. Doing it here, in prose, generated by iterating the same registry, is a natural and low-cost way to give the model a preview of the reversibility framing from Concept 3 (a read-only tool is obviously low blast-radius; `edit_file` obviously is not) before any enforcement code exists for it.

Concretely, this phase's tools section is:

```typescript
function buildToolsSection(): string {
  const lines = toolRegistry.map(
    (t) => ` - ${t.name}${t.readOnly ? " (read-only)" : ""}: ${t.description}`
  );
  return [
    `# Using your tools`,
    `You have the following tools available. Prefer them over describing what you would do — call the tool.`,
    ...lines,
    `If multiple tool calls are independent of each other, you may reason about them together, but this agent executes tool calls one at a time.`,
  ].join("\n");
}
```

`toolRegistry.map(...)` is the entire mechanism. There is no second, parallel list of tool names anywhere in `prompt.ts` — add a fourth tool to `tools.ts`'s registry and this section describes it automatically, on the very next call to `buildSystemPrompt()`, with zero edits to this file. That is the concrete, working answer to "why generate this from the registry instead of hand-writing it": the alternative is a second source of truth that Phase 2's whole `ToolDefinition` design was explicitly built to avoid (Phase 2, Concept 1).

One more precise point worth being able to state: **iterate `toolRegistry` directly here, not `getToolSchemas()`.** `getToolSchemas()` (Phase 2) deliberately strips a `ToolDefinition` down to exactly the three fields the wire protocol needs (`name`, `description`, `input_schema`) — it throws away `readOnly` on purpose, because the API's `tools` parameter has no slot for it. This phase needs `readOnly` for the prose section above, so it has to go back to the richer, unstripped `toolRegistry` array — which is exactly the seam Phase 2's own closing note called out: *"This is also the seam Phase 3 (System Prompt) will use to describe the available tools by name and behavior"* (`phase-02-tool-system.md`, `getToolSchemas()`'s doc comment).

## Concept 5: Template variables — runtime context, injected at call time

Everything in Concepts 2–4 is either fully static (identity, tone) or generated once per call from the tool registry (Concept 4) — nothing so far has depended on the filesystem or the current moment. One more category of content is neither: facts about *this specific moment* — what directory you're in, what today's date is, what branch you're on. These need to be computed inside `buildSystemPrompt()` itself, every time it's called, not baked into a string literal once at author time. Get this wrong — compute `process.cwd()` once at import time and cache it in a module-level constant, say — and every conversation your agent has for the rest of the process's lifetime reports the directory the process happened to start in, even if the user's actual working context has since changed.

**Why `.split(key).join(value)` and not `.replace(key, value)` for substitution.** The reference implementation does this substitution with `.split("{{cwd}}").join(value)` rather than the seemingly more obvious `content.replace("{{cwd}}", value)` (`claude-code-from-scratch/src/prompt.ts`, `buildSystemPrompt()`, lines 219–229). This isn't a stylistic preference — `String.prototype.replace` treats certain sequences in its *replacement* argument specially: `$&`, `$$`, `$1`, and friends are pattern tokens, not literal text, even when the search argument is a plain string rather than a regex. This is verifiable directly, independent of any documentation:

```
node -e '
const val = "line one $& more $1 stuff";
const tmpl = "before {{x}} after";
console.log("replace():", tmpl.replace("{{x}}", val));
console.log("split/join:", tmpl.split("{{x}}").join(val));
'
```

Actual output, captured while writing this tutorial:

```
replace(): before line one {{x}} more $1 stuff after
split/join: before line one $& more $1 stuff after
```

Look closely at the `replace()` line: the literal `$&` inside `val` got silently replaced with `{{x}}` (the matched text) — `replace()` interpreted `$&` in the replacement string as "insert the matched substring here," corrupting the output without an error or a warning. `split/join` has no such special-token behavior; whatever string you hand it comes out byte-for-byte unchanged. This matters concretely for exactly the values this phase injects: `getGitContext()`'s output is a real `git status --short` dump, and `git status` output can absolutely contain a literal `$` in a filename or a commit message on the `git log` line. `.split(key).join(value)` is not a stylistic quirk of the reference implementation — it's the correct tool for verbatim string substitution, and `.replace()` is the subtly wrong one.

This phase's environment section computes `cwd`, `date`, `platform`, `shell`, and git context (`getGitContext()`, calling `git rev-parse`, `git log`, `git status` via `execSync`, swallowing any error into an empty string so a non-git directory doesn't break prompt composition — adapted directly from `claude-code-from-scratch/src/prompt.ts`, lines 101–114) and builds them into a section with simple string interpolation rather than a `{{placeholder}}`-in-template-then-substitute step, since there's no need for a two-pass template when the values are computed and assembled in the same function (see Implement 3's code). The *lesson* — compute at call time, not at author/import time, and be careful about what string-substitution primitive you reach for — is the same either way.

## Concept 6: `@include`-style composition — CLAUDE.md and `.claude/rules/`

Everything so far is baked in at author time — the same text every run, for every project. Real coding agents also need a channel for *this specific project's* instructions, discovered and loaded fresh every call, without the agent's own source code needing to change per-project. That channel is a `CLAUDE.md` file (this document you're reading right now lives inside a project that has exactly this convention).

**The discovery mechanism.** Starting from the current working directory, walk *upward* toward the filesystem root, checking for a `CLAUDE.md` at every level, and collect every one found (`claude-code-from-scratch/src/prompt.ts`, `loadClaudeMd()`, lines 75–97, read directly). Files closer to the root get `unshift`'d to the front of the array — so the final concatenation order is root-to-leaf, with the CWD's own `CLAUDE.md` appended *last*. That's the same recency-effect reasoning from Concept 2: the most specific, most locally-relevant instructions should be freshest in the model's context, and "freshest" here means "closest to the end of the text," not "found first."

**`@include` — modular files instead of one giant `CLAUDE.md`.** A team's shared conventions shouldn't have to be copy-pasted into every project's `CLAUDE.md` by hand. The reference implementation supports a small include directive — a line starting with `@` followed by a path — resolved recursively:

```typescript
// adapted from claude-code-from-scratch/src/prompt.ts, lines 15-47
const INCLUDE_REGEX = /^@(\.\/[^\s]+|~\/[^\s]+|\/[^\s]+)$/gm;
const MAX_INCLUDE_DEPTH = 5;

function resolveIncludes(content, basePath, visited = new Set(), depth = 0) {
  if (depth >= MAX_INCLUDE_DEPTH) return content;
  return content.replace(INCLUDE_REGEX, (_match, rawPath) => {
    // resolve @./relative, @~/home, or @/absolute paths...
    if (visited.has(resolved)) return `<!-- circular: ${rawPath} -->`;
    if (!existsSync(resolved)) return `<!-- not found: ${rawPath} -->`;
    // ...read it, recurse into it, return the resolved content
  });
}
```

Three path forms are supported — `@./relative/to/this/file`, `@~/home/directory`, `@/absolute/path` — and two guards keep it safe: a `visited` set stops file A from including file B which includes file A (an infinite loop), and `MAX_INCLUDE_DEPTH` caps how deep a chain of includes can nest even without an actual cycle. Both failure modes degrade to an HTML comment marker in the output (`<!-- circular: ... -->`, `<!-- not found: ... -->`) rather than throwing — the same "errors are data, not an exception that kills the whole request" philosophy Phase 2, Concept 2 established for tool execution, applied here to prompt composition instead.

**`.claude/rules/*.md` — auto-loaded, no `@include` needed.** Alongside `CLAUDE.md` itself, any `.md` file dropped into a `.claude/rules/` directory at the project root gets picked up automatically, sorted by filename, each one still passed through `resolveIncludes()` in case a rule file itself references something else (`claude-code-from-scratch/src/prompt.ts`, `loadRulesDir()`, lines 51–71). This is the mechanism this very tutorial series' own repo uses in practice — the private instructions you saw injected into this conversation's context came from exactly this kind of rules-directory convention.

**What this phase deliberately does not build.** Real Claude Code discovers `CLAUDE.md` from five distinct locations (an org-wide managed policy directory, the user's home directory, the project directory walked upward, a `.local` override file, and a directory named explicitly on the command line) — `claude-code-from-scratch/docs/03-system-prompt.md`'s own simplification table calls this out directly: *"CLAUDE.md 5 层发现 + .claude 子目录"* (production) → *"从 CWD 向上遍历 + .claude/rules/"* (the teaching version), justified as *"覆盖常见场景"* — "covers the common cases" (line 296). This phase's `loadClaudeMd()` implements exactly that same trimmed-down version: upward directory walk plus one rules directory, not all five discovery locations. This is a deliberate, cited scope cut, not an oversight.

Notice `buildSystemPrompt()` is already called fresh on every `main()` invocation in this phase's `index.ts` — that's exactly the shape Phase 4's REPL needs: call it once per new session (or once per turn, if you want git status/date to stay current across a long-running session), not once at process startup and never again.

Later phases in the fuller series build directly on pieces this phase introduced without fully using yet: the `readOnly` flag surfaced in `buildToolsSection()`'s prose becomes something the loop actually *enforces on* once permissions (Phase 6) exist, and the static/dynamic prompt-caching boundary real Claude Code uses (`SYSTEM_PROMPT_DYNAMIC_BOUNDARY`, splitting the ~70% of the prompt that's identical across all users from the ~30% that's per-conversation, so the static portion can hit a shared cache) is a real, cited optimization this phase does not implement — `claude-code-from-scratch`'s own simplification table calls this out explicitly as skipped in the teaching version too, with the stated reason "a tutorial project doesn't need to optimize API cost." If cost/latency ever becomes a concern for your own build, that boundary marker is where you'd start.
